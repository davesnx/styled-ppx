# Lessons

- 2026-03-26: When asking architecture questions, avoid unexplained jargon like "token slices". Define the term in plain language first or use a concrete repo-specific example.
- 2026-03-26: Do not start implementing an architecture boundary until the data model is explicitly confirmed. Here, assuming a token-list boundary was wrong; the intended shared boundary is the existing parser AST.
- 2026-03-26: When the user says tokens should stay inside `packages/parser`, treat any token adapter in `packages/css-grammar` as the wrong direction even if it is technically incremental.
- 2026-03-26: If the parser AST still carries string payloads, avoid open-coding those string classifiers in `css-grammar`; introduce parser-owned helper/classifier functions instead.
- 2026-03-27: After landing an architectural step, remove transitional compatibility layers quickly if the user asks for the cleaner design. Do not defend scaffolding like `last_buffer` or unclassified-token adapters once the parser owns the control flow.
- 2026-03-27: When the user says "do not parse in `Rule.re`", do not preserve a token-based `Rule.re` behind an AST-to-token adapter. That still violates the requested boundary. The correct direction is for `Rule.re` itself to consume `component_value_list`/AST cursors directly, with any legacy string wrappers living outside `Rule.re`.
- 2026-03-27: Do not report `Rule.re` as fully refactored just because it consumes AST input. If `rule('a)` still threads `(input, remaining_input)` and `Pattern` still drives sequence matching/backtracking, then `Rule.re` is still a parser layer over AST and that remaining work must be called out explicitly.
- 2026-03-27: User clarified the desired shape is still `(ast => data, ast)`. The problem is not the cursor-combinator shape itself; the problem is forcing `Rule.re` to recover semantics from strings. Keep rules operating over AST tails, but push delimiter/unit/other semantic classification into `packages/parser/lib/Ast.re` and parser construction when that information is currently stringly.
- 2026-03-30: When a requested category split leaves a still-supported property without an obvious bucket, do not hide it in a generic `Legacy` module by default. Prefer a dedicated property module unless the user explicitly wants a catch-all bucket.
- 2026-03-30: When the user asks to move the obvious non-legacy leftovers out of `Legacy`, prefer dedicated property/group modules for clear modern properties (`marker`, `masonry-auto-flow`, `hyphenate-limit-*`, `initial-letter-align`, `block-overflow`) and leave only truly ambiguous/deprecated cases behind.
- 2026-03-30: If the user asks for semantic grouping of leftover `Legacy` properties, prefer small grouped modules (`Layout`, `Voice`, `Rest`, `Pause`, `Nav`, `Glyph`) instead of one-file-per-property churn.
- 2026-03-30: Property file names should match the property/group name in OcamelCase. Do not keep stale suffixes like `Experimental` in filenames once the property name is known.
- 2026-03-30: When the user wants `Values`/`Functions`/`Media` to be pure registries, do not stuff shared `%spec_module` rule definitions into those files. Split them into a dedicated shared grammar module and keep the registry files as entry lists only.
- 2026-03-30: If the user wants to remove a façade module like `Parser`, prefer flattening the library root wrapper (`Css_grammar.re`) and renaming callers, rather than relocating runtime library exports into the PPX generator layer.
- 2026-03-30: When cleaning `Shared.ml`, distinguish real shared value grammars from duplicate property ownership. Remove exact property duplicates, move `Function_*` definitions into `Functions.ml`, and keep only intentional name overlaps like shared `color` and value-grammar `position`.
- 2026-03-31: When the user explicitly asks for a single parsing path inside `packages/css-grammar`, do not preserve string-based compatibility wrappers there. Remove the duplicate API and let tests and upstream callers adapt at the boundary instead.
- 2026-03-31: Reserve `parse` for `packages/parser`. In `packages/css-grammar`, use semantic verbs like `type_check`, `validate_property`, and `infer_interpolation_types` so the API describes AST interpretation rather than source parsing.
- 2026-04-03: Avoid introducing tiny helper abstractions that hide straightforward repeated parser wrapper logic unless they clearly improve readability. In `packages/parser/lib/driver.re`, the repeated try/catch wrappers were preferred over a generic `wrap_parse` helper.
