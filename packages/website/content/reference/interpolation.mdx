# Interpolation

Interpolation is one of the most powerful and useful patterns in styled-ppx. You can use values from your code inside your styles. This is useful for reusing variables, handling themes, or conditionally applying styles based on props.

```reason
let maxWidth = CSS.px(100);

module Theme = {
  let black = CSS.black;
};

module Component = [%styled.div
  {|
  border: 1px solid $(Theme.black);
  max-width: $(maxWidth);
|}
];
```

```mlx
let max_width = CSS.px 100

module Theme = struct
  let black = CSS.black
end

module Component = [%styled.div
  {|
  border: 1px solid $(Theme.black);
  max-width: $(max_width);
|}]
```

<Callout>
  Don't confuse interpolation with [String Interpolation from Reason](https://reasonml.github.io/docs/en/primitives)
</Callout>

## Rules

`$( ... )` accepts **any expression**: identifiers, module accessors (`$(Theme.black)`), function calls (`$(darken(color, 0.2))`), or inline computations. The expression is type-checked at the interpolation site, so errors point at your source.

Interpolation works inside **declaration values** and inside **selectors** (as class references). It does *not* work on property names or at-rule preludes:

```reason
module Component = [%styled.div {|
  border: 1px solid $(Theme.black); /* ✅ value */
  &.$(active) { opacity: 1; }       /* ✅ class reference */
|}];
```

```mlx
module Component = [%styled.div {|
  border: 1px solid $(Theme.black); (* ✅ value *)
  &.$(active) { opacity: 1; }       (* ✅ class reference *)
|}]
```

It's not allowed to interpolate entire properties or half-properties, which might be slightly different from SASS/Less and JavaScript-based solutions (such as styled-components or emotion), whose string-based interpolation can be used everywhere.

**styled-ppx** forces you to be more rigid, with the promise of always being type-safe. It's the same trade-off Reason and OCaml make ❤️. Here's a valid JavaScript example `` `margin-${whatever}: 10px`{:js} `` from styled-components, which isn't valid in **styled-ppx**. The solution is simple: handle the property based on the dynamic value:

```reason
let margin = direction =>
  switch (direction) {
  | Left => [%css "margin-left: 10px;"]
  | Right => [%css "margin-right: 10px;"]
  | Top => [%css "margin-top: 10px;"]
  | Bottom => [%css "margin-bottom: 10px;"]
  };
```

```mlx
let margin = function
  | Left -> [%css "margin-left: 10px;"]
  | Right -> [%css "margin-right: 10px;"]
  | Top -> [%css "margin-top: 10px;"]
  | Bottom -> [%css "margin-bottom: 10px;"]
```

If you aren't familiar with the `[%css]{:reason}` extension, take a look [at the `[%css]{:reason}` section](./css).

## Type-safety

The expected type of an interpolation depends on its position within the property's value. Any value from any property can be interpolated:

```reason
module Size = {
  let small = CSS.px(10);
};

[%css "margin: $(Size.small)"]; // -> margin: 10px;
[%css "margin: $(Size.small) 0"]; // -> margin: 10px 0;
[%css "margin: $(Size.small) $(Size.small)"]; // margin: 10px 10px;
```

```mlx
module Size = struct
  let small = CSS.px 10
end

let _ = [%css "margin: $(Size.small)"] (* -> margin: 10px; *)
let _ = [%css "margin: $(Size.small) 0"] (* -> margin: 10px 0; *)
```

Here `margin` expects a length:

```reason
type length = [
  | `ch(float)
  | `em(float)
  | `ex(float)
  | `rem(float)
  | `vh(float)
  | `vw(float)
  | `vmin(float)
  | `vmax(float)
  | `px(int)
  | `pxFloat(float)
  | `cm(float)
  | `mm(float)
  | `inch(float)
  | `pc(float)
  | `pt(int)
  | `zero
  | `percent(float)
];
```

```mlx
type length = [
  | `ch of float
  | `em of float
  | `ex of float
  | `rem of float
  | `vh of float
  | `vw of float
  | `vmin of float

```

Since `Size.small` is `` `px(int){:reason} ``, the type-checker allows it. Interpolating, say, a `Color.t` where a length is expected is a compile error. Use the constructors from [`CSS`](./runtime) to build correctly-typed values.

## How it works: CSS custom properties

The extracted rule references a CSS custom property (`color: var(--color-<hash>)`), and the element supplies its *value* through the inline `style` attribute. Variable names are derived from your source and stable between builds. The full mechanics (naming, scoping, `@property` registration, and the positions where interpolation is rejected) live in [How it works](/how-it-works#dynamic-values-css-custom-properties).

## A note about the polymorphism of CSS

There are plenty of shorthand properties in CSS that accept different values whose meaning depends on their position, making it challenging for the type-checker to know which value to use. Some of the most problematic ones are `animation`, `box-shadow`/`text-shadow`, `background`, `transition` and `transform`.

These properties are not challenging to type because they are shorthand properties, but because they have values that are positional **<u>and</u>** optional at the same time.

Let's look at `background`:

```css
background: #fff; /* The background is white */
background: url(img.png); /* The background is an image */
background: #fff url(img.png); /* The background is white with an image */
background: url(img.png) no-repeat; /* The background is a non-repeating image */
```

To interpolate `background: $(variable1) $(variable2)`, the type-checker can't know the types of `$(variable1)` and `$(variable2)` ahead of time, because there are numerous possibilities for a valid `background` property. This is called an [overload](https://www.typescriptlang.org/docs/handbook/declaration-files/by-example.html#overloaded-functions) in other languages, and it's not available in OCaml due to its nature as a statically typed language. In those cases, split the shorthand into longhands (`background-color: $(color)`) so each interpolation has a single expected type.

## What if a property isn't supported

First, if you have the time, please [open an issue](https://github.com/davesnx/styled-ppx/issues/new). Most properties are trivial to add support for.

If time isn't your best friend, the escape hatch is a **custom property**: `--x: $(expr)` passes through unvalidated (the expression must be a `string`), and you consume it with `var(--x)`:

```reason
let value = "10px 20px";
let styles = [%css {|
  --experimental-padding: $(value);
  padding: var(--experimental-padding);
|}];
```

```mlx
let value = "10px 20px"
let styles = [%css {|
  --experimental-padding: $(value);
  padding: var(--experimental-padding);
|}]
```

Here the safety relies on you providing a valid CSS string.

For a general overview of property support, take a look at [support for CSS](/css-support).

## Not valid interpolation

The extension payload itself must be a [string literal](./css#validation): the CSS has to be known at compile time for [static extraction](/how-it-works) to parse, validate, and extract it.

```reason
/* 🔴 Can't pass a variable reference */
let value = "display: block";
module X = [%styled.div value];
```

```mlx
(* 🔴 Can't pass a variable reference *)
let value = "display: block"
module X = [%styled.div value]
```

Dynamic *values* go through `$( ... )` interpolation; dynamic *structure* goes through the language: pattern matching to different `[%css]` blocks, composition with `CSS.merge`.
