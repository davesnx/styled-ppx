# CSS syntax

styled-ppx parses string CSS in different modes, depending on the extension you use.

| API | Mode | What it accepts |
| --- | --- | --- |
| `%css`, `%styled.tag` | Declaration block | Declarations plus nested selectors and nested at-rules |
| `%styled.global` | Stylesheet | Top-level rules and at-rules exactly as in CSS |
| `%keyframe` | Keyframes | `from` / `to` / percentage step blocks |

Every mode is parsed, validated and type-checked at compile time: unknown properties and invalid values are compile errors (with "did you mean" suggestions for typos).

## Declaration blocks

`%css` and `%styled.tag` use the same declaration-block grammar.

You write the body of a rule, not the outer selector. styled-ppx generates the class selector for you and scopes nested rules to it.

```css
display: grid;
gap: 16px;

&:hover {
  gap: 20px;
}

svg path {
  fill: currentColor;
}

@media (min-width: 768px) {
  gap: 24px;
}
```

In this mode you can mix:

- plain declarations
- nested selectors
- nested at-rules such as `@media`, `@supports`, and `@container`

Declarations can appear before or after nested blocks, although keeping declarations first is still the clearest style.

## Stylesheets

`%styled.global` parses a real stylesheet.

That means top-level selectors stay exactly as written and are not attached to a generated class.

```css
html, body {
  margin: 0;
  padding: 0;
}

@media (prefers-reduced-motion: reduce) {
  * {
    animation: none;
  }
}
```

Use this mode for global resets, document-level selectors, `@font-face`, and top-level at-rules. Bare declarations at the top level are rejected — they must live under a selector. See [Global styles](./globals).

## Selectors and nesting

Declaration blocks support nested selectors. The parser accepts both `&`-based nesting and bare nested selectors.

### `&` means "the current generated selector"

Use `&` when you want to append to, combine with, or reposition the current selector.

```css
&:hover { color: red; }
&::before { content: ""; }
&.active { opacity: 1; }
&#hero { scroll-margin-top: 80px; }
& > span { display: inline-block; }
& + & { margin-top: 8px; }
div & { margin-left: auto; }
```

Typical expansions look like:

- `&:hover` -> `.generated:hover`
- `&.active` -> `.generated.active`
- `div &` -> `div .generated`
- `& + &` -> `.generated + .generated`

### Bare nested selectors become descendants

You do not need `&` for every nested selector.

```css
.child { color: blue; }
#selected { font-weight: 600; }
div { padding: 0; }
svg path { fill: currentColor; }
svg * { vector-effect: non-scaling-stroke; }
[data-open] { opacity: 1; }
```

These become descendant selectors relative to the generated class, for example:

- `.child` -> `.generated .child`
- `div` -> `.generated div`
- `svg path` -> `.generated svg path`

One exception: a nested selector cannot *start* with a bare pseudo-class or pseudo-element (`:hover { ... }` without `&`). Per CSS Nesting semantics that would be a descendant selector, which is rarely what you mean — styled-ppx rejects it:

```
Bare leading pseudo selector `:hover` is ambiguous in nested CSS. Per CSS
Nesting Level 1 §3.1 it descendant-joins with the enclosing selector
(producing `<parent> :hover`), which matches descendants rather than the
element itself. Write `&:hover` for compound (`<parent>:hover`, the usual
intent), or `& :hover` to opt into the explicit descendant form.
```

### Supported selector features

The supported syntax includes the usual selector building blocks:

- type selectors: `div`, `button`, `svg path`
- class, id, and universal selectors: `.child`, `#main`, `*`
- attribute selectors: `[disabled]`, `[type="text"]`, `[href^="https"]`
- pseudo-classes: `:hover`, `:focus`, `:target`, `:not(...)`, `:is(...)`, `:where(...)`, `:has(...)`
- pseudo-elements: `::before`, `::after`, `::first-line`, `::first-letter`
- combinators: descendant, `>`, `+`, `~`
- comma-separated selector lists

## Class references in selectors

Interpolating a **binding** in selector position references another `[%css]` value's generated classes. The reference is resolved statically — the actual class names are substituted into the extracted stylesheet:

```reason
let outlined = [%css "border: 1px solid"];

let button = [%css {|
  padding: 8px;

  &.$(outlined) {
    padding: 7px;
  }

  & .$(outlined) {
    margin: 4px;
  }
|}];
```

```mlx
let outlined = [%css "border: 1px solid"]

let button = [%css {|
  padding: 8px;

  &.$(outlined) {
    padding: 7px;
  }
|}]
```

- `.$(name)` splices the binding's classes as a compound (`.cssA.cssB` when the binding has several atoms). Bare `$(name)` (no leading `.`) is accepted only when the binding has a single atom — otherwise the ppx raises ``Bare `$(name)` selector interpolation expanded to N class names; this position only accepts a single class. Prefix with `.` to use a class chain instead: `.$(name)`.``
- References work inside combinators, pseudo-classes (`:not(&.$(other))`, `:has(.$(other))`), comma lists, and inside nested `@media` / `@supports` bodies.
- Same-file bindings resolve during ppx expansion; **cross-module** references (`$(Card.container)`) resolve when [the stylesheet is generated](/how-it-works#class-references-in-selectors). Referencing a binding that doesn't exist is a compile error; references across *library* boundaries are unsupported (see the [build errors](/getting-started/dune-setup#troubleshooting) they produce).

## Media queries and other at-rules

Declaration blocks support nested at-rules, especially:

- `@media`
- `@supports`
- `@container`

Example:

```css
display: grid;

@media (min-width: 768px) {
  grid-template-columns: repeat(2, 1fr);
}

@supports (display: grid) {
  gap: 16px;
}

@container card (width >= 400px) {
  padding: 24px;
}
```

At-rules nest under selectors (and vice versa) and render correctly scoped, e.g. `&:hover` inside `@media` becomes `@media (...) { .generated:hover { ... } }`.

**At-rule preludes cannot be interpolated.** `@media (max-width: $(breakpoint))` is a compile error:

```
Interpolation in @media preludes is not supported during static extraction.
CSS custom properties (var()) are not valid in media query conditions.
Inline the value directly.
```

Write the condition literally; if a breakpoint truly must vary, define one block per variant and switch between them in code.

If you need keyframes, use the dedicated [keyframe extension](./keyframe) instead of writing them inline in a declaration block.

## Interpolation placement

`$(...)` accepts any OCaml/Reason expression and is supported in two places:

- **declaration values** — type-checked from the property it appears in, lowered to a CSS custom property:

  ```css
  color: $(textColor);
  border-bottom: 1px solid $(borderColor);
  height: calc(100vh - $(topMenuHeight));
  ```

- **selectors** — a static [class reference](#class-references-in-selectors) to another binding:

  ```css
  &.$(activeClass) { opacity: 1; }
  ```

Not supported (compile errors with an explanation):

- at-rule preludes: `@media (max-width: $(bp))` (see [above](#media-queries-and-other-at-rules))
- inside `url(...)` — the error reads: ``Interpolation inside `url(...)` is not supported: browsers don't substitute `var()` there. Inline the URL: `url("/path/to/asset")`. Or interpolate the whole value: `src: $(font_src)`.``
- value interpolation under selectors that escape the element's subtree (e.g. `& + .sibling { color: $(c) }`). Inline custom properties are set on `&` and only inherit downward, so a sibling or a `:not(&)`/`:has(&)` subject could never read the value; the ppx rejects the declaration instead of emitting CSS that silently never applies: ``Cannot interpolate into the value of `color` under `& + .sibling`: the selector targets an element outside `&`'s subtree [...] Instead, target `&` or a descendant, or write a literal value or a globally-inherited theme `var(--...)` directly.``

See [Interpolation](./interpolation) for the type-checking rules.

## Edge cases and caveats

- The final semicolon is optional for the last declaration.
- Declaration blocks also recover many missing semicolons before nested selectors or nested `@media` blocks, including after interpolated values, but explicit semicolons are still recommended.
- Declaration-block APIs do not need an outer wrapper like `.foo { ... }`; stylesheet APIs (`%styled.global`) do.
- If a property is unsupported by the type-checker, the escape hatch is a custom property: `--my-value: $(expr)` passes through unvalidated (the expression must be a `string`), and you can consume it with `var(--my-value)`.

## Examples by API

### `%css` / `%styled.tag`

```css
padding: 16px;

  padding: 20px;
}

```
