# CSS coverage oracle

Status: the oracle is implemented, and all four unification items are
done: (1) the registry dump owns the at-rule classification, (2) the
website support tables are generated, (3) the conformance tests derive
from the oracle, and (4) the css3test vendored data and the MDN URL
checklist are retired.

## What exists

styled-ppx now measures its CSS coverage against a machine-readable oracle
instead of estimating it:

- **Vendored spec inventory.** `packages/css-grammar/data/webref-css.json`,
  a slim index (names, value-definition syntax, spec ids) derived from
  [`@webref/css`](https://github.com/w3c/webref), MIT-licensed data
  machine-extracted from W3C specifications. Refreshed manually with
  `make css-oracle-update` (the only step that needs network); committed so
  nothing else does.
- **Registry dump.** `packages/css-grammar/bin/registry_dump.exe` prints
  every name the grammar layer knows (properties, shared values, functions,
  media-query grammars) plus the media/container feature-name inventories
  used by at-rule prelude validation.
- **Coverage report.** `packages/css-grammar/data/coverage.md`, generated
  by `make css-oracle` (`scripts/css-oracle/report.mjs`). A summary table,
  explicit missing lists for properties, legacy aliases, functions and
  media features, and an at-rule status table.
- **Website support tables.** `packages/website/src/generated/css-support-tables.mdx`,
  generated by the same `make css-oracle` run
  (`scripts/css-oracle/support-page.mjs`). User-facing tables (coverage
  summary, per-spec-module property coverage, at-rule support, media
  features) imported as an MDX component by
  `packages/website/content/css-support.mdx`. Both render scripts share
  one derivation chain via `scripts/css-oracle/data.mjs`; the website
  build consumes the committed fragment and never depends on dune.
- **Spec-conformance suite.** `packages/ppx/test/css-support/spec-conformance.t`,
  generated by the same `make css-oracle` run
  (`scripts/css-oracle/conformance.mjs`). A cram test that turns the
  coverage report into an executable claim: every registry-supported
  property must compile with the simplest keywords-and-zero-values
  expansion of its @webref/css value-definition syntax (`supported.re`,
  one `dune build`), and every spec property the registry does not
  support must fail to compile (`unsupported.re`, checked via
  `dune describe pp` so the harness asserts every declaration rather
  than only the first error). Properties whose syntax has no such
  expansion are listed in the generated file header and left to the
  hand-written suites; registry bugs (spec-legal values the grammar
  rejects) can be parked on a documented exception list in
  `conformance.mjs`. Runs as part of `make test-css-support`.
- **CI gate.** `make css-oracle-check` regenerates all three artifacts
  and fails if any differs from the committed one, so registry changes
  keep the report, the website, and the conformance suite honest and
  spec-inventory refreshes are deliberate, reviewable diffs.

The workflow:

```
make css-oracle           # regenerate all oracle artifacts from committed data
make css-oracle-update    # refresh webref-css.json from npm, then regenerate
make css-oracle-check     # CI: fail when any artifact is stale
```

Closing a gap = adding a `[%spec_module]` to `packages/css-grammar` (plus
its runtime witness), rerunning `make css-oracle`, and committing the
report diff. The report syntax strings come straight from the specs, but
styled-ppx's spec dialect diverges from raw CSS Value Definition Syntax in
places (interpolation alternatives, runtime witnesses), so treat them as a
starting point, not paste-ready input.

## Unified coverage data sources

Two inventories remain, with code as the only styled-ppx source of
truth:

| Source | Location | Consumed by | Nature |
|---|---|---|---|
| Grammar registry | `packages/css-grammar` (code) | the compiler | ground truth |
| Webref inventory | `data/webref-css.json` | coverage report, website tables, conformance suite | spec ground truth |

Everything else derives from those two via `make css-oracle`. The
retired sources: the hand-vendored css3test data
(`packages/ppx/test/generate-css-support/data.js`, replaced by the
generated spec-conformance suite) and the MDN URL checklist
(`data/reference.txt`, whose checklist role `coverage.md` covers).

The unification plan, all landed:

1. **The registry dump is the single "supported" inventory.** Done:
   `registry_dump.exe` emits the at-rule support classification from
   `packages/parser/lib/At_rules.re`, the same sets `Css_file.re` and
   `Resolve.re` consume, which removed the one hand-maintained map
   (`report.mjs`'s old at-rule table) from the oracle.
2. **The website support page becomes generated.** Done:
   `scripts/css-oracle/support-page.mjs` emits
   `packages/website/src/generated/css-support-tables.mdx` from the same
   loaders as `coverage.md` (shared `scripts/css-oracle/data.mjs`);
   `css-support.mdx` imports it as an MDX component. The hand-written
   prose stays, the tables come from the registry, and
   `make css-oracle-check` gates the fragment alongside `coverage.md`.
3. **Conformance tests derive from the oracle, not from css3test.** Done:
   `scripts/css-oracle/conformance.mjs` generates
   `packages/ppx/test/css-support/spec-conformance.t` from the same
   loaders as `coverage.md` (shared `scripts/css-oracle/data.mjs`). For
   every registry-supported property, a compiles-test built from the
   webref syntax string's simplest expansion: a small value-definition
   syntax parser picks bare keywords from `|`/`||` alternatives, `0` for
   `<length>`/`<number>`/`<integer>` (range permitting), `0%` for
   `<percentage>`, omits `?`/`*`, and takes minimum occurrences for
   `+`/`#`/`{m,n}`; properties needing anything fancier are skipped and
   listed in the generated header. For every missing property, an
   expect-error test. This replaced the aging `data.js` vendored copy
   (which nothing executed) and turned the coverage report into an
   executable claim; the suite is committed, runs under
   `make test-css-support`, and `make css-oracle-check` gates drift.
   Spec-legal values the registry rejects can be parked on the documented
   `compileExceptions` list in `conformance.mjs`. Each entry is a known
   css-grammar bug and a candidate fix; the list is empty since the 17
   bugs it surfaced were fixed (issue #592).
4. **Retire `data/reference.txt`.** Done, together with
   `packages/ppx/test/generate-css-support/`; `coverage.md` covers the
   checklist role.
